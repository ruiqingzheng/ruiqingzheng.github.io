    我们先从 jwt 的使用开始 ,

    https://github.com/auth0/node-jsonwebtoken

    然后再来jwt 的 原理

    主要思想就是部分加密, token = header + payload + 签名

    header 和payload 都只是base64编码基本就等于是明文, 然后把header 和 payload (也就是我们需要传递的信息) 一起进行加密, 得到的就是签名

    那么这个就是jwt token , 可见它只有签名是真正加密的, payload 是需要传递的信息这部分只是编码了一下

    但这样部分加密且明文携带信息的token , 依然是安全地, 因为服务器端在获取到客户端传递的token 后 , 会verify 这个签名, 只要携带的信息被更改, 那么就会抛出异常Invalid Signature

    更具体的说就是

    服务器端存在有签名时候设置的安全字符串 , 而用户上传的token 中有payload , 于是服务器端根据payload , header 还有这个secret(安全字符串) 重新加密 , 比对签名就能知道信息是否被篡改

    https://jwt.io/ 测试体会下消息字段被修改后, 签名验证就会失败

    注意：secret是保存在服务器端的，jwt的签发生成也是在服务器端的，secret就是用来进行jwt的签发和jwt的验证，所以，它就是你服务端的私钥，在任何场景都不应该流露出去。一旦客户端得知这个secret, 那就意味着客户端是可以自我签发jwt了。

    使用中间件来验证

    // check-auth

    const jwt = require("jsonwebtoken");

module.exports = (req, res, next) => {
  try {
    const token = req.headers.authorization; // 新版本express 直接可以获取头部
    const decoded = jwt.verify(token, process.env.JWT_KEY);
    req.userData = decoded;
    next();
  } catch (error) {
    return res.status(401).json({
      message: "Auth failed"
    });
  }
};

    // router 中使用中间件

    在上一部中, 中间件已经把 jwt.verify 返回的解码后的数据赋值给了 req.userData

    所以在后面的处理请求中, 可以直接获取 req.userData

    # signature

    jwt的第三部分是一个签证信息，这个签证信息由三部分组成：

    • header (base64后的)
    • payload (base64后的)
    • secret

    这个部分需要base64加密后的header和base64加密后的payload使用.连接组成的字符串，然后通过header中声明的加密方式进行加盐secret组合加密，然后就构成了jwt的第三部分。

    // javascript
var encodedString = base64UrlEncode(header) + '.' + base64UrlEncode(payload);

var signature = HMACSHA256(encodedString, 'secret'); // TJVA95OrM7E2cBab30RMHrHDcEfxjoYZgeFONFh7HgQ

    将这三部分用.连接成一个完整的字符串,构成了最终的jwt:
