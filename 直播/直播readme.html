                                            # 视频编码

                                            -- 视频编码的主要作用是将视频像素数据（RGB，YUV 等）压缩成为视频码流，从而降低视频的数据量。如果视频不经过压缩编码的话，体积通常是非常大的，一部电影可能就要上百 G 的空间。视频编码是视音频技术中最重要的技术之一。视频码流的数据量占了视音频总数据量的绝大部分。高效率的视频编码在同等的码率下，可以获得更高的视频质量。

                                            当前使用最多的视频编码方案就是 H.264 也叫做 AVC 实际中使用最多的就是 x264 了，性能强悍（超过了很多商业编码器），而且开源 Google 推出的 VP8 属于和 H.264 同一时代的标准。总体而言，VP8 比 H.264 要稍微差一点

                                            # 音频编码

                                            -- 音频编码的主要作用是将音频采样数据（PCM 等）压缩成为音频码流，从而降低音频的数据量。音频编码也是互联网视音频技术中一个重要的技术。但是一般情况下音频的数据量要远小于视频的数据量，因而即使使用稍微落后的音频编码标准，而导致音频数据量有所增加，也不会对视音频的总数据量产生太大的影响。高效率的音频编码在同等的码率下，可以获得更高的音质。

                                            常见的就是 aac, mp3, wma 音频编码技术近期绝大部分的改动都是在 MP3 的继任者——AAC 的基础上完成的。

                                            # 封装格式

                                            -- 封装格式的主要作用是把视频码流和音频码流按照一定的格式存储在一个文件中

                                            常见封装格式有: avi, mp4, mov, flv

                                            但 avi 不支持流媒体,流媒体指的就是即可以“边下边播”

                                            一般一种封装格式能支持多种视频编码格式

                                            有些格式更“万能”一些，支持的视音频编码标准多一些，比如 MKV。而有些格式则支持的相对比较少，比如说 RMVB

                                            比如 MP4 封装格式就支持多种视频编码格式 MPEG-2, MPEG-4, H.264, H.263等

                                            一般可以通过视频文件的属性来查看视音频技术 , 也就是上面提到的 3 个方面,

                                            封装格式: 比如 mov, mp4, flv 等 Video 编码: AVC 也就是 H264 Audio 编码: AAC

                                            # 直播相关视音频参数

                                            直播服务普遍采用了 RTMP 作为流媒体协议，FLV 作为封装格式，H.264 作为视频编码格式，AAC 作为音频编码格式。采用 RTMP 作为直播协议的好处在于其被 Flash 播放器支持。而 Flash 播放器如今已经安装在全球 99%的电脑上，并且与浏览器结合的很好。因此这种流媒体直播平台可以实现“无插件直播”，极大的简化了客户端的操作。封装格式，视频编码，音频编码方面，无一例外的使用了 FLV + H.264 + AAC 的组合。FLV 是 RTMP 使用的封装格式，H.264 是当今实际应用中编码效率最高的视频编码标准，AAC 则是当今实际应用中编码效率最高的音频编码标准。视频播放器方面，都使用了 Flash 播放器。

                                            注: 这只是 pc 端, 在移动端不可能用浏览器结合 flash 来做客户端, 一般都是用的 librtmp 来收流然后用 sdk 提供的播放器来播放

                                            # rtmp 协议

                                            rtmp 的服务器端用 nginx 即可, 或者 srs 它本质上也还是用 nginx

                                            直播数据流的推送和接收 ffmpeg 可以进行推流和收流, 还有转发流, 而且很多播放器都是在 ffmpeg 的基础上二次开发

                                            但 ffmpeg 是控制台方式, 所以才有一些其他更方便的图形界面工具软件, 它最适合用来做教学直播, 或者教学视屏录播,

                                            但不适合移动开发直播 app

                                            因为 rtmp 形式的直播流, 无法使用 web 浏览器来播放, 它必须使用第三方 sdk 或者其他工具软件

                                            而 hls 则可以使用 web 来接收直播流, 这样就对移动端的开发降低难度同时也更灵活, 可以使用 web 技术来开发直播客户端

                                            配置

                                            /usr/local/etc/nginx/nginx.conf

                                            可配置的服务有, rtmp, hls, http/flv

                                            http/flv 和 rtmp 相似

                                            // open nginx.conf -a code , 用 code 打开 nginx.conf

                                            rtmp {
    server {
        listen 1935;
        chunk_size 4000;

        # rtmp conig
        application rtmplive {
            live on;
            max_connections 1024;
        }

        # hls config

        application hls {
            live on;
            hls on;
            hls_path /usr/local/var/www/hls;
            hls_fragment 5s;
        }
    }
}

                                            修改 nginx.conf 中的 server 块, 添加 /hls 路径

                                            
location /hls {
    types{
        application/vnd.apple.mpegurl m3u8;
        video/mp2t ts;
    }
    root /usr/local/var/www;
    add_header Cache-Control no-cache;
}

                                            // 注意, 配置 nginx http 访问的 root 地址只需要 /usr/local/var/www 即可, 而不是 /usr/local/var/www/hls

                                            推流的时候实际是推送到 /usr/local/var/www/hls , hls 是 rtmp 中的 server 部分配置的应用名

                                            修改完配置后重启 nginx -s reload

                                            # hls

                                            rtmp 配置段中配置了 application hls

                                            hls_path 是存储视频分片的目录 /usr/local/var/www/hls, 后面我们在推流的时候就实际上是推送到了这个目录

                                            nginx 中又配置了application hls 我们又在 http 中配置了 /hls 访问路径实际访问的就是 hls_path

                                            于是, 直播的分片视频就能被 http://localhost/hls/$streamName.m3u8 这个地址来访问了

                                            比如, 推送:

                                            ffmpeg -re -i test.mp4 -vcodec libx264 -acodec aac -f flv rtmp://localhost:1935/hls/stream

                                            收流 用 safari 打开 http://localhost:8080/hls/stream.m3u8 即可访问上面的推流

                                            经测试发现 hls 的延时有点过分

                                            录屏推流的流程：

                                            1. 客户端原始帧编码为 H264 裸流（MediaCodec，Android API） —>
                                            2. 再封装为 FLV 格式的视频流（Java） —>
                                            3. 按照 rtmp 流媒体协议通过 librtmp（JNI + C）推流到 RTMP 流媒体服务器 —>
                                            4. 客户端播放器解码观看

                                            注意：

                                            Demo 省略了 librestreaming 中的 OpenGL 处理帧率的过程，也就意味着我们使用的是 MediaCodec 直接编码后的数据，并没有 OpenGL 绘制 VirtualDisplay 映射给 MediaCodec.createInputSurface 中 Surface 的这个过程，目的是将流程简单化。OpenGL 方面的使用之后我也会介绍说明。

                                            录屏数据流的获取

                                            自己测试的录屏项目录屏主要用的是 MediaRecorder 这个类, 它很方便的可以用来录屏保存为文件,

                                            而我们推流的话, 是需要得到 H264 裸流, 而 MediaRecorder 使用中并没有直接接触到裸流

                                            在更早版本 andriod 中, 如果不用 MediaRecorder 的话, 录屏保存文件用的是 MediaMuxer 这个类, 其他步骤是一致的

                                            录屏的过程中核心类是 MediaProjection , 其中的 createVirtualDisplay 方法,

                                            Creates a VirtualDisplay to capture the contents of the screen.

                                            而这个方法中的, 需要传递 surface 参数, surface 是数据消费者 , 也就是说录屏的数据输出传递给该 surface

                                            The surface to which the content of the virtual display should be rendered

                                            所以 createVirtualDisplay 传参 mediaRecorder.getSurface() 即可把录屏流数据传递给 mediaRecorder 对象中的 surface

                                            // MediaRecorder 录屏流程

                                            先设置一下 mediaRecorder 的参数 --> 再调用 createVirtualDisplay --> mediaRecorder.start();

                                                        mediaRecorder.setOutputFile(videoUri);
            mediaRecorder.setAudioSource(MediaRecorder.AudioSource.MIC);
            mediaRecorder.setVideoSource(MediaRecorder.VideoSource.SURFACE);

            mediaRecorder.setVideoFrameRate(30);

            mediaProjection.createVirtualDisplay("MainActivity", DISPLAY_WIDTH, DISPLAY_HEIGHT, mScreenDensity,

            mediaRecorder.start();

                                            如果不用 MediaRecorder 的方式, 那么通过 MediaProjection 的 createVirtualDisplay 方法传递不同的 surface 参数就能获取到流数据

                                            核心用到的类还有 VideoEncoder ,

                                            首先通过配置初始化 mVideoEncoder 对象 , 比如视频编码格式 H264 就是初始化该对象可以传递的参数

                                            传递 surface 给录屏方法获取输入流, mMediaProjection.createVirtualDisplay(...mVideoEncoder.getInputSurface())

                                            获取输出流 mVideoEncoder.getOutputBuffer(index); 这里得到的就是编码后的视频缓存, 也就是 H264 裸流

                                            如果只是录屏保存文件的话, 那么调用 MediaMuxer 这个类, https://www.jianshu.com/p/aeadf260258a

                                            可见不使用 MediaRecorder 录屏的话, 流的编码和输入输出, 需要的是 mVideoEncoder

                                            如果要推流就必须要 VideoEncoder 来获取 H264 裸流

                                            流程

                                            1. 初始化 VideoEncoder, 设置编码格式 -->
                                            2. 调用 mMediaProjection.createVirtualDisplay 设置流输入到 mVideoEncoder 的 surface
                                            3. mVideoEncoder 获取裸流 mVideoEncoder.getOutputBuffer(index)
                                            4. MediaMuxer 将裸流写入到文件 mMuxer.writeSampleData(track, encodedData, buffer);
